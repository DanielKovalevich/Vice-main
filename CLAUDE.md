# CLAUDE.md — Vice architecture and maintenance guide

Vice is a Medal.tv-style instant-replay clip recorder for Linux. A daemon keeps
a rolling capture buffer; pressing a global hotkey saves the last N seconds as
a clip. Clips get a local gallery UI and optional public share links.

## Process model

Vice runs as up to three cooperating processes:

1. **Daemon** (`vice start`, entry `vice/main.py:ViceDaemon`). Owns the
   recorder child process, the hotkey listener, the share/UI HTTP servers, and
   Discord Rich Presence. Usually launched by the systemd user service
   (`vice.service`, generated by install.sh) or by vice-app.
2. **Recorder child**: gpu-screen-recorder (GSR) by default, spawned and
   supervised by the daemon. GSR holds the replay buffer in memory and flushes
   it to a file when it receives SIGUSR1.
3. **Desktop app** (`vice-app`, `vice/app.py`). A pywebview window
   (QtWebEngine preferred, WebKit2GTK fallback, browser last resort) pointed at
   the daemon's local HTTP UI. Closing the window leaves the daemon recording.
   vice-app starts the daemon if it is not running and restarts it after
   package upgrades (version mismatch check).

IPC between app and daemon is a line-oriented Unix socket at
`/tmp/vice/vice.sock` (`status`/`stop`/`clip` commands); the PID file is
`/tmp/vice/vice.pid`. Everything else goes over HTTP/WebSocket.

## Module map (`vice/`)

| Module | Role |
|---|---|
| `main.py` | CLI (click) + `ViceDaemon` orchestration: startup/shutdown, hotkey bindings, clip/session flows, live config apply, Discord presence loop, `vice doctor` diagnostics, uninstall. |
| `recorder.py` | Recording backends behind the `Recorder` ABC: `GSRRecorder` (replay buffer via SIGUSR1) and `SegmentRecorder` (30 s rolling segments for wf-recorder/ffmpeg). Also: GSR command construction, audio/display option listing for the UI, clip trimming/watermarking, clip naming (`Vice_Clip_N[_Game].{mp4,mkv}`). |
| `media.py` | Shared ffprobe helpers. `probe_media` prefers the container (format) duration because fragmented MP4 (what GSR writes) has no per-stream duration tags. Also temp-file cleanup for interrupted in-place edits. |
| `share.py` | Two aiohttp servers. Local (127.0.0.1:`sharing.port`): web UI, `/api/*`, WebSocket. Public (0.0.0.0:`sharing.public_port`, default port+1): only `/c/{slug}` (embed page), `/v/{slug}` (video), `/t/{slug}` (thumbnail). Public never exposes the API. Cloudflare quick tunnel (cloudflared) provides the public URL; there is deliberately no other tunnel fallback. |
| `hotkey.py` | evdev-based global hotkeys (works on X11, any Wayland compositor, TTY). A supervisor rescans `/dev/input` every 3 s so keyboards can unplug/replug. Single vs double-tap dispatch (double-tap toggles session recording). Requires read access to `/dev/input/event*` (udev uaccess / `input` group, handled by install.sh). |
| `app.py` | vice-app desktop launcher: daemon lifecycle, webview backend selection + Chromium stability flags, JS bridge (`quit_app`, `copy_to_clipboard`, ...), launch-failure dialogs that surface the daemon's real stderr. |
| `config.py` | Dataclasses persisted as `~/.config/vice/config.toml`. Unknown keys are dropped on load; the settings API merges partial updates and validates hotkeys strictly. |
| `runtime.py` | Environment repair for systemd-launched daemons: fixes unexpanded `${HOME}`-style values, recovers `WAYLAND_DISPLAY`/`XDG_RUNTIME_DIR` from systemd user env or socket scanning. |
| `active_window.py` | Focused-window adapters (X11/xdotool, Hyprland/hyprctl, Sway/swaymsg) returning `{"process", "class", "pid"}`. KDE/GNOME Wayland are unsupported by design (no stable API). |
| `discord_rpc.py` | Minimal Discord IPC client for Rich Presence. |
| `audio.py` | Clip-saved notification sound (paplay/aplay/ffplay). |
| `ui/` | The web UI: static HTML/CSS/JS served by share.py, no build step, no framework. State lives in `ui/scripts/state.js` globals; WebSocket messages drive updates (no HTTP polling loops). |

## The clip flow (GSR, the default path)

1. Daemon binds hotkeys: single-tap = clip, double-tap = session toggle,
   plus per-duration preset keys (`hotkeys.clip_presets`).
2. `GSRRecorder.start()` spawns
   `gpu-screen-recorder -w <target> -f <fps> [-s WxH] -r <buffer> -c <container> [-k codec] [-a tracks...] -o <out_dir>`.
   If GSR exits within 1 s, its stderr is parsed into a readable error and
   startup fails loudly.
3. On clip: SIGUSR1 → GSR writes the buffer to a new file in the output dir.
   `save_clip` waits for the file to appear, then `_wait_for_finalized_clip`
   waits for write **inactivity** (not a fixed total deadline — long buffers on
   slow disks can take minutes). Probe success requires `media.probe_media`.
4. The file is renamed to `Vice_Clip_N[_Game].{ext}` (game tag from the
   curated games list via `clip_tag_cb`, runs in a thread), trimmed to the
   requested duration (`-c copy` first, re-encode fallback), optionally
   watermarked, then `_emit` notifies the daemon → share server broadcast →
   UI toast.
5. Session recordings are a separate single-file recording (always mp4) with
   highlight timestamps stored as JSON sidecars in
   `~/.local/share/vice/highlights/`.

The SegmentRecorder path (wf-recorder/ffmpeg, opt-in via config) keeps a ring
of 30 s segments in `/tmp/vice/segs` and concatenates the needed window on
clip. `container`/`audio_tracks` are GSR-only; segment backends log a warning
and stay single-track mp4.

## Policies (deliberate decisions — do not "fix" these)

- **GSR is the only auto backend.** wf-recorder/ffmpeg are never auto-selected;
  install.sh aborts if GSR cannot be installed. No silent fallback recording.
- **cloudflared or a visible error.** A serveo SSH fallback once produced junk
  share links and MITM'd traffic; tunnel failure now broadcasts `tunnel_error`
  to the UI instead of trying something else.
- **Game detection is curated-list-only** (bundled `data/games.json` + user
  `discord.custom_games`). Window titles are never matched heuristically and
  never leave the machine (Discord presence and filename tagging both follow
  this).
- **The destructive moov remux is guarded.** `share._remux_moov` only replaces
  a clip when the remuxed copy probes sane and is of comparable size. It once
  truncated healthy clips to 0.02 s; keep the guard.
- **Errors surface, never swallow.** Every failure path should log why and,
  where a user is waiting, reach the UI (WS broadcast, dialog, or doctor
  output). vice-app captures the daemon's pre-logging stderr to
  `vice-daemon-stderr.log` so even import-time crashes appear in the error
  dialog.
- **Docs/user-facing copy**: no em-dashes, no marketing fluff; match the
  README's voice.

## Error-handling conventions

- Subprocess failures: capture stderr, summarize with
  `_summarize_process_error`, raise `RuntimeError` with the readable message.
- Background tasks log exceptions (`log.exception`) instead of dying silently.
- `except Exception: pass` is forbidden; minimum is a `log.debug` with the
  reason (grep for `log.debug` in `share.py` helpers for the pattern).
- Anything hardware/compositor-specific that cannot be tested in CI gets a
  comment explaining the workaround and the issue number.

## Tests

```bash
python -m unittest discover tests/
```

No external services needed. ffmpeg/ffprobe-dependent tests generate tiny real
fixtures and skip when ffmpeg is absent. The share-server tests bind real
sockets on localhost. `tests/test_install_script.py` and `test_ui_static.py`
are static assertions over install.sh and the UI files (cheap regression
guards for shell/HTML details). Version sync across `vice/__init__.py`,
PKGBUILD, and .SRCINFO is enforced by `test_cli.py` — bump all of them
together on release.

## Packaging / install

- **AUR** (`vice-clipper`, PKGBUILD in repo): system site-packages, binaries
  in /usr/bin.
- **install.sh** (everything else): detects distro (pacman/apt/dnf/zypper),
  installs system deps, builds GSR from a pinned tag as the user (sudo only
  for `meson install`; GSR 5.12.5 is auto-picked on FFmpeg 6 systems like
  Ubuntu 24.04), creates a venv at `~/.local/share/vice/venv` with
  `--system-site-packages`, symlinks `~/.local/bin/vice{,-app}`, writes the
  systemd user service and desktop file. rpm-ostree systems (Bazzite etc.)
  are rejected early with an explanation; Flatpak is the intended path there.
- The systemd unit passes display env vars via `PassEnvironment`; Hyprland/
  Sway users may need `systemctl --user import-environment` in compositor
  config (documented in README troubleshooting).

## Known platform constraints

- Active-window detection (game tagging, Discord presence): X11, Hyprland,
  Sway only.
- QtWebEngine runs with `--disable-accelerated-video-decode` (hardware video
  decode renders black/grey on many drivers) and `--disable-features=Vulkan`
  on NVIDIA (the GBM-less Vulkan fallback segfaults). Override with
  `QTWEBENGINE_CHROMIUM_FLAGS`, append with `VICE_WEBVIEW_FLAGS`.
- Discord inlines only mp4 under ~50 MB; quick-tunnel URLs change on daemon
  restart.
- wf-recorder needs `wlr-screencopy-unstable-v1`; GNOME/KDE portals are only
  reachable through GSR's portal capture.
